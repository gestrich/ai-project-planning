Skill

plan

The project-aware entry point for "where are we and what's next?" — and the reconciliation entry point for "what's drifted, what should graduate, what's stale?"

Reads
The project's AGENTS.md, plus whichever planning sources it declares (Jira, Confluence, local plans/, Slack-as-context).
Writes
Nothing directly. Routes write requests to per-source reference documents, which gate writes behind explicit confirmation.
Configuration
The ## Project Planning section in AGENTS.md — see Configuration.

What it does

The plan skill doesn't assume any particular planning system. It reads the project's AGENTS.md to find out which sources of truth the project uses, then loads only the reference documents that match — keeping context lean and avoiding patterns that don't apply.

  1. Reads AGENTS.md at the project root and locates the project-planning section.
  2. Identifies which planning sources apply — any combination of Jira, Confluence, the local plans/ folder, or a Slack channel for context.
  3. Loads only the matching reference documents from plugin/skills/plan/:
    • jira.md — Jira read/write discipline
    • confluence.md — Confluence read-before-write discipline
    • slack.md — Slack as read-only context capture
    • local.md — the plans/ folder convention
  4. Synthesizes a short, source-cited view, grouped by area (UI, infra, telemetry, …), surfacing what's awaiting you at the top.

When to reach for it

Default mode — synthesize a current view:

  • "what's the plan", "what's the plan for X"
  • "what's next", "what should I be looking at"
  • "show me the plans", "where are we on Y", "what's the status of Z"
  • "catch me up on this project", "give me the rundown"

Reconciliation mode — compare local plans/ against external sources:

  • "sync plans", "reconcile with Jira", "reconcile the docs"
  • "what's drifted", "what should graduate", "what's stale in plans/"
  • "is the local plan still aligned with Jira/Confluence"

Default mode

A good plan response is:

  • Source-cited. Every statement traces to Jira (with issue keys), Confluence (with page link), the local plans/ file path, or the Slack message permalink. If a fact has no source, the skill flags it as inference.
  • Grouped by area, not by source. You think in terms of workstreams; the skill pulls from all sources into one coherent view, then cites per item.
  • Surface what's awaiting you. Decisions, blockers, asks aimed at you — those go at the top.
  • Honest about gaps. If a source is stale or a Jira epic hasn't been touched in three weeks, the skill says so rather than fabricating motion.

If AGENTS.md has no project-planning section, the skill says so plainly and offers to run bootstrap instead of guessing the planning shape from the repo.

Reconciliation mode

Reconciliation answers three questions, side-by-side:

  1. Graduation candidates. Items in plans/ that are firm enough they should live in the external source of truth now.
  2. Drift. Cases where plans/ and the external source disagree about the same item — a Jira issue marked Done while the local plan still has it "in flight", etc.
  3. Stale local items. Bullets in plans/ that the external source has overtaken — an idea that's now shipped, a question answered in a Confluence page, a workstream deprioritized in Jira.

The report is grouped by these three questions, not by source — that's how you think about hygiene. Every actionable item shows up; the skill doesn't trim to a "top N" because under-reporting hides the work.

If plans/ is the project's sole source of truth (no external system declared), there's nothing to reconcile against — the skill says so and stops. Same if no plans/ exists in the first place.

Writes are gated

The plan skill itself doesn't perform mutations. When you ask for a write that touches a planning source — updating a Confluence page, transitioning a Jira issue, deleting stale local content — it defers to the reference document for that source, each of which carries the appropriate read-before-write or explicit-authorization rules. In reconciliation mode each acted-on item is its own confirmation; a single "apply everything" is too coarse for reports that often span 10+ items across multiple sources.
