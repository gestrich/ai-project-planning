Skill

bootstrap

The on-ramp. Surveys an existing project, proposes a plan-of-changes, and applies it after you confirm.

Reads
AGENTS.md, existing planning folders, commit history (for Jira/Confluence/Slack hints), git remote -v.
Writes
Creates notes/, plans/, sprints/ as needed; uses git mv for any file relocations; edits or creates AGENTS.md; symlinks CLAUDE.md → AGENTS.md.
Configuration
None to start — this skill is what produces the configuration.

What it does

The skill runs in three phases. They don't collapse — you see the proposed plan before any edits happen.

1. Survey

Read-only project inspection:

  • Determine the project root (usually cwd; asks if ambiguous).
  • Check for AGENTS.md and any existing project-planning section (follows the CLAUDE.md → AGENTS.md symlink).
  • Check for existing notes/, plans/, sprints/ folders.
  • Check for stray planning artifacts — a top-level NOTES.md, TODO.md, planning/ directory, dated transcript files at the root.
  • Best-effort source-of-truth hints (never asserts — these become questions to ask): Jira issue keys in commit messages, Confluence URLs in docs, Slack archive links, GitHub remote(s).

2. Propose

Produces a written plan-of-changes inline, covering folders to create, files to move (always via git mv to preserve history), AGENTS.md changes, planning sources to declare, GitHub repos to declare, and — most importantly — the plugin skills the project should use, each with a one-line "what it's for / when to reach for it" note.

Ask any clarifying questions inline before finalizing. Common ones: "is this Jira project the source of truth or just a place issues get filed?", "do you want plans/ as the canonical plan or for in-incubation work?", "this commit references ABC-123 — is that the project key you want declared?"

3. Apply

Only after explicit confirmation — "looks good", "go ahead", "apply", "do it". Creates folders, git mvs relocated files, edits or creates AGENTS.md (with the CLAUDE.md symlink if needed), and reports the file paths it touched.

When to reach for it

Run it the first time you bring a project into the plugin, and re-run it when the planning shape changes (a new Slack channel, a new repo, a new doc source).

/bootstrap set this project up for the planning plugin
/bootstrap onboard this repo
/bootstrap refresh — I added a new Slack channel and a new repo

If the project has already been bootstrapped, the skill triggers as a refresh. It surfaces what's there, proposes deltas, and avoids duplicating content.

The planning content it produces in AGENTS.md

See the full shape in Configuration. Free-form markdown — the skill picks a heading that fits your existing AGENTS.md, doesn't force a particular section name. What lands in the content:

  • Which planning sources apply — Jira project keys, Confluence space and root page IDs, the local plans/ folder, and which is the source of truth for shipped work.
  • The project's Slack channel, if any — name and ID, flagged for context — so the other skills know where to read team discussion.
  • The project's GitHub repos, in owner/repo form.
  • The plugin skills the project uses, each with a one-line note on what it's for and when to reach for it. This gives any future session immediate orientation without needing to read each SKILL.md upfront.
