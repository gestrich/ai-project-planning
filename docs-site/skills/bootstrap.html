Skill

bootstrap

The on-ramp. Surveys an existing project, proposes a plan-of-changes, and applies it after you confirm.

Reads
AGENTS.md, existing planning folders, commit history (for Jira/Confluence/Slack hints), git remote -v.
Writes
Creates notes/, plans/, plans/sprints/ as needed; uses git mv for any file relocations; edits or creates AGENTS.md; symlinks CLAUDE.md → AGENTS.md.
Configuration
None to start — this skill is what produces the configuration.

What it does

The skill runs in three phases. They don't collapse — you see the proposed plan before any edits happen.

1. Survey

Read-only project inspection:

  • Determine the project root (usually cwd; asks if ambiguous).
  • Check for AGENTS.md and any existing project-planning section (follows the CLAUDE.md → AGENTS.md symlink).
  • Check for existing notes/, plans/, plans/sprints/ folders.
  • Check for stray planning artifacts — a top-level NOTES.md, TODO.md, planning/ directory, dated transcript files at the root.
  • Best-effort source-of-truth hints (never asserts — these become questions to ask): Jira issue keys in commit messages, Confluence URLs in docs, Slack archive links, GitHub remote(s).

2. Propose

Produces a written plan-of-changes inline, covering folders to create, files to move (always via git mv to preserve history), AGENTS.md changes, planning sources to declare, GitHub repos to declare, and — most importantly — the plugin skills the project should use, each with a one-line "what it's for / when to reach for it" note.

Ask any clarifying questions inline before finalizing. Common ones: "is this Jira project the source of truth or just a place issues get filed?", "do you want plans/ as the canonical plan or for in-incubation work?", "this commit references ABC-123 — is that the project key you want declared?"

3. Apply

Only after explicit confirmation — "looks good", "go ahead", "apply", "do it". Creates folders, git mvs relocated files, edits or creates AGENTS.md (with the CLAUDE.md symlink if needed), and reports the file paths it touched.

When to reach for it

  • "bootstrap this project", "bootstrap the project for planning"
  • "set up project planning here", "wire this repo up to the planning skills"
  • "make this project work with the planning plugin"
  • "onboard this repo", "get this project ready for planning"
  • Landing in a fresh repo and asking what you need to add so the other skills will work.

If the project has already been bootstrapped, the skill still triggers — but as a refresh. It surfaces what's there, proposes deltas (e.g., a Slack channel you've started using but haven't declared), and avoids duplicating sections.

The planning content it produces in AGENTS.md

See the full shape in Configuration. Free-form markdown — the skill picks a heading that fits your existing AGENTS.md, doesn't force a particular section name. Two things matter most:

  • It enumerates which planning sources apply, so planning knows which reference documents to load.
  • It lists the plugin skills the project uses, each with a one-line note on what it's for and when to reach for it. This gives any future session immediate orientation without needing to read each SKILL.md upfront.

What this skill does not do

  • It does not write to external systems. No Jira project creation, no Confluence page creation, no Slack channel setup — only writes to the local repo.
  • It does not run other skills as part of bootstrap. After applying the plan, you invoke planning, docs-update, etc. yourself.
  • It does not enforce a folder layout beyond the conventions named in the plugin. If a project already uses a different layout and wants to keep it, the skill proposes declaring that in AGENTS.md instead of moving files.
