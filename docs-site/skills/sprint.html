Skill

sprint

The scrum-master capability: helps you pick what to do this week given everything that's already in motion. Surfaces decisions; doesn't make them.

Reads
Recent files in notes/, the previous sprint file in plans/sprints/, the configured planning sources (via plan), and the Slack channel declared in AGENTS.md (read-only, last week).
Writes
plans/sprints/YYYY-MM-DD.md (start of week) after explicit confirmation.
Configuration
Same as plan — uses whatever's declared in AGENTS.md.

What it does

The sprint skill is a coach, not an automated planner. It gathers context, asks the questions that need answering before a sprint can be decided, and only writes a sprint file once you've signed off on the draft.

  1. Resolve the sprint start date with date +%Y-%m-%d (Monday of the current or upcoming week — asks you if it's ambiguous).
  2. Gather context: recent notes/ files, the configured plan sources (delegating to plan's reference documents), the most recent file in plans/sprints/, and a week of Slack activity from the configured channel.
  3. Coach. Surface the implicit decisions in any candidate sprint shape, then ask the challenging questions — which workstream is most at risk of slipping, what's been carried over and is it still right, what's surfaced in notes that hasn't been triaged, what's awaiting a decision you can make right now, what's not going in and are you okay with that.
  4. Draft the sprint file inline. Path: plans/sprints/<YYYY-MM-DD>.md.
  5. Confirm explicitly, then write. Edits are cheap; surprise files are not.

When to reach for it

  • "what should I do next", "what should I focus on this week"
  • "what's on deck", "what's load-bearing right now", "what should I prioritize"
  • "plan my sprint", "let's plan the sprint", "next sprint", "kick off the new sprint"
  • Anything that asks for a focused, time-boxed view of the coming week's work, even if the word "sprint" isn't used.

If you're asking for a current-state synthesis ("what's the plan", "where are we"), that's plan. sprint is specifically about picking what to do next.

Sprint file shape (loose)

The skill encourages this shape; it doesn't enforce it.

# Sprint: <YYYY-MM-DD>

## On deck

### <Domain — e.g. UI>
- [ ] item, with a one-line *why* and a source link (Jira key, Confluence page, plans/ file, or note path)

### <Domain — e.g. infra>
- [ ] ...

### Personal / cross-cutting
- [ ] ...

## Carried over
- [ ] items from last sprint still in flight, with a note on what's blocking them

## Decisions awaiting Bill
- the choice that's gating something on this sprint

## Out of scope this week
- things considered and explicitly deferred (so the next sprint isn't confused about why they're missing)

Domains come from the project itself — the skill reads the project's plans/ domain documents or AGENTS.md to discover them. It doesn't invent new domains the project hasn't already named.

What this skill does not do

  • It does not write to external systems. No Jira transitions, no Confluence edits, no Slack posts.
  • It does not modify notes/ or domain documents in plans/. It reads them.
  • It does not enforce a sprint cadence or template. Two-week sprints, custom shapes — the skill follows your lead.
  • It does not auto-roll items from the previous sprint. Carrying something over is a decision, not a default.
