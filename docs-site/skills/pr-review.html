Skill

pr-review

Documentation-aware GitHub PR review. Fetches a PR, cross-references the diff against the project's documented conventions, and surfaces findings in both directions: PR-side and docs-side.

Reads
The GitHub repo(s) declared in AGENTS.md (via the gh CLI when available, GitHub REST API otherwise); the documentation sources declared in AGENTS.md (Confluence pages, local docs, plans/).
Writes
Posts PR comments via gh pr review/gh pr comment only after explicit confirmation. Stages doc edits with the same discipline as docs-update.
Configuration
Requires at least one GitHub repo declared in AGENTS.md. A documentation source is strongly recommended — without one the skill can only summarize the diff with no documented baseline to compare against.

What it does

A PR-side finding is "the diff diverges from what the docs say should happen." A docs-side finding is "the diff introduces a pattern or decision the docs don't reflect yet." The two are produced together. A PR that adopts a new pattern isn't necessarily wrong; sometimes the docs are stale. Reporting both sides lets you decide which way to close the gap.

The skill runs in three phases: fetch → cross-reference → propose. Everything before phase 3 is read-only.

1. Fetch

  • Resolve the repo from AGENTS.md. If you give a full PR URL, the skill takes the repo from the URL and confirms it appears in AGENTS.md — if it doesn't, asks once (the PR may belong to a different project).
  • Use the gh CLI when available. Fall back to the GitHub REST API only when gh isn't installed.
  • Pull the PR description, the changed-file list, the diff, and the head SHA. The head SHA is captured so any later comment doesn't end up on a stale diff.
  • No local checkout in v1. Intentional — keeps the skill portable and avoids dragging in build/test machinery.

2. Cross-reference

For each meaningful change in the diff, look up what the project's documentation says about that area. The documentation sources are whichever ones AGENTS.md declares:

  • Confluence pages — via planning/confluence.md (Atlassian MCP schemas loaded via ToolSearch, fetched as markdown for inspection).
  • Local docs — files inside the project, typically under docs/ or plans/.
  • Plan files — plans/plan.md and the domain documents it references.

The goal isn't to mechanically grep for filenames mentioned in the docs. For each load-bearing change in the diff, the skill asks: is there documented guidance that speaks to this, and does the change match it? Renamed local variables don't trigger a lookup; a new public API, a changed architectural boundary, a new dependency, a different error-handling pattern, a deviation from a naming convention — those do.

3. Propose

Two clearly-separated sections, shown inline.

A. PR review against documented conventions

For each PR-side finding:

  • What the divergence is — the specific change, the file/line range, what the docs say should happen instead.
  • Where the doc lives — the page, file, or section being relied on.
  • Severity — how load-bearing the divergence is. The skill grades; it does not mark everything as critical.
  • Suggested resolution — change the PR, update the doc, or accept the divergence as a one-off with a comment in the PR explaining why.

If the diff doesn't diverge from anything documented, the skill says so plainly. An empty PR-side section is a valid outcome.

B. Documentation update suggestions

For each docs-side finding: what the docs are missing, where it should land, and why it's worth landing. This output shape intentionally mirrors docs-update so the two skills compose cleanly in a weekly cadence.

3a. Act (only after explicit confirmation)

  • PR comments — drafted inline first. Before posting, re-check the head SHA; if it's moved, the skill surfaces that and re-fetches rather than commenting on a stale diff.
  • Doc edits — same write discipline as docs-update. Confluence reads use the read-before-write protocol strictly. In-repo doc edits show the proposed diff and wait for confirmation.

When to reach for it

  • "review this PR", "review PR #123", "what do you think of this PR"
  • "is this PR aligned with our docs", "does this match our conventions"
  • "check the open PRs for doc drift", "sweep open PRs"
  • Pasting a bare GitHub PR URL into the conversation when AGENTS.md declares that repo.

If you're asking for a generic code review without doc cross-referencing — "look for bugs in this PR" — that's a different job; this skill is specifically the documented-conventions read.

What this skill does not do

  • It does not check out the PR locally in v1. Diffs come from the GitHub API.
  • It does not run tests, linters, or builds. The PR's own CI handles that.
  • It does not post PR comments or edit docs without explicit confirmation. Read-before-write applies to both directions.
  • It does not generate generic code review feedback (style nits, micro-optimizations, subjective preferences). Findings must trace back to something the project's docs actually say, or to a docs-shaped gap the PR exposes.
  • It does not modify local notes/ or plans/.
