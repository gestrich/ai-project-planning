Skill

docs-update

Analyzes recent team discussion against the project's documentation and proposes updates you can act on. Reads where discussion happens and where the docs live from your AGENTS.md; follows whatever update workflow you've described there.

What it does

  1. Reads the discussion source declared in AGENTS.md over a recent window (default: last seven days).
  2. Reads the current documentation source declared in AGENTS.md.
  3. Analyzes the two together and proposes updates — clarifications, missing answers, recurring questions that deserve a permanent home, decisions made in chat that the docs don't yet reflect.
  4. Hands off the action however AGENTS.md says updates happen in this project. Always with your confirmation.

Try: "update the docs", "what's missing from the docs?", "review last week's discussion for doc updates", "what should we have written down?"

What it reads from AGENTS.md

  • Where team discussion happens. Usually a Slack channel; could be Discord, GitHub Discussions, a mailing list, or anything else.
  • Where the docs live. A Confluence space, an in-repo folder, a wiki, a Notion workspace, an S3 bucket — wherever the project keeps them.
  • How docs get updated. Free-form prose. Projects do this in wildly different ways — direct edit access, a PR-reviewed in-repo folder, an issue against another team's repo, a request to a docs team. Whatever the workflow is, write it in AGENTS.md and the skill will follow it.

Without a named discussion source the skill has nothing to read. Without a named docs source it can still report what came out of discussion, but framed as "things the team is talking about that don't appear to be written down anywhere." Without a clear update workflow, the skill will ask before doing anything.

What the proposals look like

Each proposed update carries:

  • What the update is — a clarification, a missing answer, a recurring question, a decision needing a permanent home.
  • Where it should land — the specific doc/page/section if known. If ambiguous, surfaced as a question.
  • Why it's worth landing — the discussion signal that prompted it (link if available).

Common patterns the skill flags:

  • A question got asked more than once in different threads → the answer deserves a permanent home.
  • A decision was made but the doc still describes the prior approach → the doc needs an update.
  • A concept the team is using freely doesn't appear in the docs at all → it needs to be introduced.
  • The doc is correct but ambiguous enough that people keep asking the same clarifying question → tighten the wording.

One-off questions that got one-off answers don't count. The skill uses judgment about what's recurring vs. transient.

How the action happens

The skill doesn't bake in an update workflow — your AGENTS.md describes one. Whatever it says, the skill follows. If you can edit the docs directly, it stages the edit using the right protocol for that source. If updates flow through another team or another process, it produces the artifact that workflow calls for (a message, an issue, a PR comment) and hands it to you to send.

Each acted-on proposal is its own explicit ask. The proposal list is not the confirmation.

When to reach for it

  • "update the docs", "what's missing from the docs"
  • "review last week's discussion for doc updates"
  • "what should we have written down"
  • A recurring weekly check on whether the documented state still matches reality.

If you're asking a one-shot question that just happens to need docs context ("what does the spec say about X"), that's plans in status mode. docs-update is the sweep — discussion in, doc proposals out.
