Skill

docs-update

Closes the loop between team discussion (usually Slack) and canonical documentation (usually Confluence). Reads recent activity and proposes updates you can act on.

Reads
The Slack channel declared in AGENTS.md (last week, read-only) and the documentation source declared in AGENTS.md (Confluence pages, an in-repo docs/ folder, etc.).
Writes
When the project owns the docs: stages a Confluence edit (with read-before-write discipline) or an in-repo doc edit after explicit confirmation. When another team owns the docs: drafts a short message to that team — does not send it.
Configuration
Requires a Slack channel and a documentation source declared in AGENTS.md. Without those, the skill says so and stops.

What it does

The skill runs in three phases: gather → propose → stage. Gather and propose are read-only; nothing leaves the project until you confirm.

1. Gather

  • Resolve the time window (default: last seven days; use date +%Y-%m-%d to anchor it). Honor a different window if you name one.
  • Read the Slack channel named in AGENTS.md, read-only — load Slack MCP schemas via ToolSearch, pull load-bearing threads, resolve user IDs to names, capture permalinks.
  • Read the documentation source declared in AGENTS.md — Confluence (with the read-before-write discipline), or in-repo docs. Don't skip this step; proposing updates without re-reading what the docs currently say is how you propose duplicates and contradictions.

2. Propose

Produces a structured list of proposed updates. Each item carries:

  • What the update is — a clarification, a missing answer, a recurring question, a decision needing a permanent home.
  • Where it should land — the specific doc/page/section. If ambiguous, surfaces as a question.
  • Why it's worth landing — the Slack permalink that prompted it.
  • Who owns the doc — the project (we can stage the write) or another team (we draft a message instead).

Good proposals usually fall into a few shapes:

  • A question got asked more than once in different threads → the answer deserves a permanent home.
  • A decision was made in a thread but the doc still describes the prior approach → the doc needs an update.
  • A concept the team is using freely in chat doesn't appear in the docs at all → it needs to be introduced.
  • The doc is correct but ambiguous enough that people keep asking the same clarifying question → tighten the wording.

One-off questions that got one-off answers don't count. The skill uses judgment about what's recurring vs. transient.

3. Stage

  • Project owns the doc. Offers to stage the edit. For Confluence, follows the read-before-write discipline: read live HTML, capture the version, compose the update as a delta, surface unexpected content, write back with the version for optimistic concurrency. For in-repo docs, shows the proposed diff inline and waits for confirmation.
  • Another team owns the doc. Drafts a short message — what's changed, the Slack permalink, a concrete suggested edit. Hands it to you for the actual send. This skill does not post.

The proposal list is not the confirmation. Each acted-on item is its own explicit ask.

When to reach for it

  • "update the docs", "what should be in the docs", "what's missing from the docs"
  • "review last week's Slack for doc updates", "sweep Slack for doc updates"
  • "what should we have written down", "anything to land in Confluence from Slack"
  • A recurring weekly check on whether the documented state still matches reality.

If you're asking a one-shot question that just happens to need docs context ("what does the spec say about X"), that's planning in status mode. docs-update is the sweep — discussion in, doc proposals out.

What this skill does not do

  • It does not post to Slack. Drafting a message to another team is fine; sending it is not this skill's job.
  • It does not write to documentation without explicit confirmation. Read-before-write applies to every body-content mutation.
  • It does not summarize the channel for catch-up purposes — that's planning in status mode (with Slack as a context source).
  • It does not modify local notes/ or plans/.
