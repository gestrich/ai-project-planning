Skill

docs-update

Analyzes recent project activity against the project's documentation and proposes updates you can act on. Reads what to scan, where docs live, and how updates happen from your AGENTS.md — and follows whatever workflow you've described.

Why this skill exists

A project usually has documentation in more than one shape:

  • Project docs — process, status, decisions, plans. The kind of thing that lives on a wiki or a Confluence space.
  • Technical docs associated with code repositories — API references, architecture notes, conventions, runbooks. Often inside the repo itself; sometimes elsewhere.

Both go stale. Decisions get made in chat and never written down. PRs change a public API but the docs that describe it don't move. Keeping these in sync with reality is a chore. This skill is the chore: scan recent activity, scan the current docs, and propose what to update.

What it does

  1. Reads the activity sources declared in AGENTS.md over a recent window (default: last seven days). Activity can be team discussion (a Slack channel, etc.), merged PRs in associated code repos, or anything else the project considers signal for doc staleness.
  2. Reads the documentation sources declared in AGENTS.md — both project docs and any repo-associated technical docs.
  3. Analyzes them together and proposes updates: clarifications, missing answers, recurring questions that deserve a permanent home, decisions made in chat that the docs don't yet reflect, technical details that no longer match the code.
  4. Hands off the action however AGENTS.md says updates happen. Always with your confirmation.

Try: "update the docs", "what's missing from the docs?", "review last week's discussion for doc updates", "are our docs in sync with the code?", "any doc gaps from recent PRs?"

What it reads from AGENTS.md

  • Activity sources to scan. Any combination: team discussion (Slack channel, Discord, GitHub Discussions); one or more code repositories whose recent PRs and commits are worth scanning for technical-doc drift; anything else the project considers signal.
  • Documentation sources to compare against. Usually a mix — high-level project docs (Confluence, a wiki, Notion) and repo-associated technical docs (an in-repo docs/ folder, a README, generated API references). AGENTS.md should say which docs cover what.
  • How docs get updated. Free-form prose. Projects do this in wildly different ways — direct edit access, a PR-reviewed in-repo folder, an issue against another team's repo, a request to a docs team. Whatever the workflow is, write it in AGENTS.md and the skill will follow it.

Without a named activity source the skill has nothing to scan. Without a named docs source it can still report what surfaced from activity, framed as "things showing up that don't appear to be written down anywhere." Without a clear update workflow, the skill will ask before doing anything.

What the proposals look like

Each proposed update carries:

  • What the update is — a clarification, a missing answer, a recurring question, a decision needing a permanent home, a technical detail that no longer matches the code.
  • Where it should land — the specific doc/page/section if known. If ambiguous, surfaced as a question.
  • Why it's worth landing — the signal that prompted it (a discussion link, a PR link, a commit).

Common patterns the skill flags:

  • A question got asked more than once → the answer deserves a permanent home.
  • A decision was made but the doc still describes the prior approach → the doc needs an update.
  • A concept the team is using freely doesn't appear in the docs at all → it needs to be introduced.
  • A merged PR changed a public API, an architectural boundary, or a convention, and the technical docs still describe the old shape → the docs need to be brought current.
  • The doc is correct but ambiguous enough that people keep asking the same clarifying question → tighten the wording.

One-off questions, tiny PRs that didn't change documented behavior, and noise commits don't count. The skill uses judgment about what's recurring or load-bearing.

How the action happens

The skill doesn't bake in an update workflow — your AGENTS.md describes one. Whatever it says, the skill follows. If you can edit the docs directly, it stages the edit using the right protocol for that source. If updates flow through another team or another process, it produces the artifact that workflow calls for (a message, an issue, a PR comment, a draft) and hands it to you.

Each acted-on proposal is its own explicit ask. The proposal list is not the confirmation.

When to reach for it

  • "update the docs", "what's missing from the docs"
  • "review last week's discussion for doc updates"
  • "are our docs in sync with the code", "any doc gaps from recent PRs"
  • A recurring weekly check on whether the documented state still matches reality.

If you're asking a one-shot question that just happens to need docs context ("what does the spec say about X"), that's plans in status mode. docs-update is the sweep — activity in, doc proposals out.
