Configuration

A project opts into the plugin by adding a ## Project Planning section to its AGENTS.md. Free-form markdown — no enforced schema. The skills read it in natural language.

Where it lives

AGENTS.md sits at the project root and is the standard cross-runtime instruction file (Claude Code reads CLAUDE.md; keep it as a symlink to AGENTS.md so both runtimes find the same content).

The ## Project Planning section is one section among many; the rest of AGENTS.md remains whatever the project already has.

A worked example

Use this as a starting point. Drop subsections that don't apply; add project-specific notes where they help.

## Project Planning

This project uses the [project-planning plugin](https://github.com/gestrich/ai-project-planning).

### Sources of truth

- **Jira**: project key `EXAMPLE`, primary epic `EXAMPLE-100`. Issues here are authoritative for shipped work.
- **Confluence**: space `EX`, root page id `123456`. Long-form plans and decisions live here.
- **Local `plans/`**: in-incubation ideas and personal items. Not authoritative for shipped work.
- **Slack**: `#example-team` (id `C0123456789`) is the canonical channel — read-only context.

### GitHub repos

- `acme-org/example-service` — primary backend
- `acme-org/example-web` — web client

### Skills used by this project

- `notes` — dump raw voice transcripts into `notes/`. Use when I paste a brain-dump.
- `plan` — synthesize what's in motion across Jira, Confluence, and `plans/`. Use when I ask "what's the plan" or "where are we on X".
- `sprint` — pick what to do this week given recent notes, Jira, and the previous sprint file. Use when I ask "what should I do next".
- `bootstrap` — re-run if the planning shape changes (new Slack channel, new repo, etc.).
- `docs-update` — sweep last week of the Slack channel for things that should land in Confluence. Use weekly.
- `pr-review` — review PRs against the docs declared above. Use when I paste a PR URL.

### Project-specific notes

The web client mirrors the backend's REST contract; PR reviews should check both sides when an endpoint changes.

What each subsection does

Sources of truth

Tells plan which reference documents to load. A Jira project key triggers plan/jira.md; a Confluence space and root page id trigger plan/confluence.md; a Slack channel id (marked for context) triggers plan/slack.md; a mention of plans/ as a source triggers plan/local.md.

Mark which source is authoritative for shipped work. When both plans/ and an external system are listed, the plugin treats the external system as the source of truth and plans/ as supplementary unless told otherwise.

GitHub repos

Used by pr-review to know which repos belong to the project. List them in owner/repo form. The first repo is usually the one the project is checked out as.

Skills used by this project

The most important subsection. Enumerate each plugin skill the project uses, with a one-line what it's for / when to reach for it note. This gives any future session immediate orientation without needing to read each SKILL.md upfront, and lets skills check the list before firing on ambiguous triggers.

Omit skills the project doesn't use. Don't pad.

Project-specific notes

Anything project-specific the skills should know — cross-repo invariants, naming conventions, default sprint cadence, whether the docs are owned by another team, etc.

Read-by-default discipline

Every skill in the plugin reads freely but never writes without explicit confirmation. That applies to local file edits as well as Jira transitions, Confluence edits, Slack sends, and PR comments. The proposal list a skill shows you is not the confirmation — each individual action is.

Folder layout in the project

Folder layout lives in the project, not the plugin:

your-project/
├── AGENTS.md            # contains the section above
├── notes/               # raw transcripts (created by `notes`)
└── plans/               # local planning artifacts
    ├── plan.md          # manifest pointing at the other plan files
    └── sprints/         # one file per sprint, dated by start of week

The bootstrap skill creates these folders and the AGENTS.md section in a single guided pass.
